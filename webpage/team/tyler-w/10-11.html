Reliable Data Transfer

A reliable data transfer is used with a reliable channel, where no transferred data bits are corrupted and all are delivered in the order in which they were sent.

TCP is a reliable data transfer protocol, however it is implemented on top of an unreliable end-to-end network layer (IP).

Building a Reliable Data Transfer Protocol

Reliable Data Transfer over a Perfectly Reliable Channel

Finite-State Machine: an abstract machine that can be in one of a finite number of states.

There are seperate FSM's for the sender and the receiver, and they each have one state.

The sending side of the data transfer accepts data from the upper layer via the rdt_send(data) event, creates a packet containing the data via the make_pkt(data) event, and sends the packet into the channel.

On the receiving side, the data transfer receives a packet from the underlying channel via the rdt_rcv(packet) event, and removes the data from the packet via the action extract (packet, data).

Reliable Data Transfer over a Channel with Bit Errors

The message dictation protocol uses positive and negative acknowledgements, which let the sender know what has been received successfully, and what has been received in an error. Reliable data transfer protocols based on this retransmission are known as Automatic Repeat reQuest (ARQ) protocols.

Three additional protocol capabilities are required in ARQ protocols to handle the presence of bit errors:

  • Error detection: Requires extra bits be sent along with the data for detection and correction (i.e. UDP uses Internet checksum field)
  • Receiver feedback: Because the sender and receiver are typically very distant from each other, it requires explicit feedback from the receiver. (i.e. positive (ACK) and negative (NAK) acknowledgement replies)
  • Retransmission: A packet that is received in error at the receiver will be retransmitted by the sender.
