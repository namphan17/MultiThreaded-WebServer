  1. How does reliable data transfer work?

    • A protocol, using reliable data transfer, must account for packet loss, as well as corrupted data.

    • There are multiple ways data can be corrupted, either because of delay or because of network errors. Packet loss could occur multiple ways as well, with either router queues being full, or the router connection being down.

    • So, this protocol, in order to ensure that data was received in full, would have to check a checksum, and send an acknowledgement to the sender to let that host know that a retransmission isn't needed.

    • If the message is garbled due to packet corruption, then the receiving host would send a NACK, or a non-acknowledgement, to the sender. This would indicate that the sender would need to retransmit the same packet again.

    • If the sender doesn't receive an acknowledgement after a set amount of time, the estimated delay, then they sender will retransmit the packet regardless of information received by the receiver.

  2. How does TCP differ from another reliable data transfer system?
    • TCP uses a SYN-ACK 3 way handshake to send data, that is the sender requests a connection, the receiver acknowledges the connection, and the sender responds with another acknowledgement with data following.

    • Similarly, TCP uses a FIN-ACK handshake at the end of the data transfer to terminate the connection that was originally initiated.

    Increasing the speed of data transfer


  3. Go-Back-N Method

    • Increases data transfer by allowing the data to constantly flow through N amount of packets, sending an acknowledgement, or NACK if necessary, when N amount of packets are done being transferred.

  4. Selective-Repeat

    • Send acknowledgement per packet, and get retransmission from host when a packet wasn't received. Allowing for faster data flow due to the fact that the sender may only need to send one packet, whereas Go-Back-N would have the sender retransmit all packets after the failed one.
